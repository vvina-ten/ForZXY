Beginner's Guide

Behavior Trees & py_trees

How a robot decides what to do — built from small, simple blocks.

1 · The big idea

A behavior tree is a way to organize a robot's "brain." Instead of writing one giant if/else, you break the job into small blocks — some blocks check things, some blocks do things — and you arrange those blocks in a tree.

This keeps the logic easy to read, easy to change, and easy to reuse.

2 · What is a "tick"?

A tick is one round of running the tree. Each tick, the robot starts at the top (the root) and works its way down, checking and doing as it goes. The tree is ticked again and again, many times per second.

3 · Three return states

Every block, after it runs, reports one of three results:

SUCCESSThe check or action worked this time.
FAILUREThe check or action did not work this time.
RUNNINGStill working — keep going next tick.
RememberSome actions don't finish in one tick. While they're still busy, they keep returning RUNNING.

4 · How to read the diagrams

Condition and Action are both ovals — tell them apart by colour: yellow = check, blue = do.

5 · Control blocks

Sequence — do it in order

A Sequence runs its children one by one. If a child fails, the whole sequence stops there. So every step must pass to reach the end.

Robot exampleStand steady → make sure the ball is at your foot → kick it. If the robot isn't standing, it never tries to kick.

Selector — try until one works

A Selector tries its children in order until one succeeds. The first child that works wins, and the rest are skipped. Great for "main plan, then backup."

Robot exampleAre you standing? If yes, carry on. If not, run the fall-recovery routine.

Parallel — run together

A Parallel pushes several children at the same time, then decides the overall result by a rule (for example, succeed when all succeed, or when one succeeds).

Robot exampleTrack the ball with your head while watching the battery level.

Memory — resume or restart?

Sequences and Selectors have a memory setting. It decides what happens on the next tick when a child was still RUNNING.

With memory

Next tick, jump straight back to the child that was RUNNING and skip the earlier ones. Good for a long chain of tasks you don't want to re-check every time.

Without memory

Next tick, start again from the first child and re-check every condition. Good when earlier conditions can change and you want the tree to react.

Robot exampleIn the line-following tree below, suppose FollowLine is RUNNING. With memory, the next tick goes straight back to FollowLine. Without memory, the next tick re-checks IsRobotStanding and HasLine first — so if the robot just fell over, it notices right away.
RememberWith memory = resume (skip ahead). Without memory = reactive (re-check from the top).

6 · Condition blocks

A Condition only checks something and answers yes (SUCCESS) or no (FAILURE). It should not change anything — it just reads the state. (Drawn as a yellow oval.)

Examples: IsRobotStanding, HasTarget, IsLineDetected (does the camera see the line on the floor?).

RememberCondition = "is it true?" It looks, it doesn't touch.

7 · Action blocks

An Action actually does the work — move, turn the head, send a command. An action may finish in one tick, or take many ticks (returning RUNNING until it's done). (Drawn as a blue oval.)

Examples: StartWalking, RunAction("kick"), FollowLine (keep steering based on where the line is).

RememberAction = "do it." This is the block that can stay RUNNING.

8 · Decorator blocks

A Decorator wraps one child and changes how it behaves. It doesn't replace the child — it adds a rule on top. (Drawn as a diamond.)

Inverter — flip the answer

An Inverter turns SUCCESS into FAILURE and the other way around.

Robot exampleInverter(HasObstacleAhead) succeeds when there is no obstacle ahead — so the robot is allowed to keep moving.

Retry / Repeat — try again, or do it again

Retry runs the child again after a failure (up to N times). Repeat runs it again after a success (N times).

Robot exampleRetry(FindBall, 3) — if the robot doesn't find the ball this time, search a few more rounds before giving up.

Status remapping advanced

Some decorators don't change what the child does — they only change the status it reports upward. These are built into py_trees, for example SuccessIsRunning, FailureIsRunning, and SuccessIsFailure.

Robot exampleSuccessIsRunning(HeadScanOnce) — even after one scan succeeds, the parent is told "still searching," so this branch keeps getting ticked. This turns a one-shot action into ongoing behavior.
One rule for decoratorsThe child does the work; the decorator changes how the child is read.

10 · Five lines to remember

Cheat sheet

Sequence — do it step by step; one wrong step and it stops.

Selector — try them one by one; one that works is enough.

Condition — only checks, doesn't act.

Action — really runs, really moves.

Decorator — same block, changed rule.
